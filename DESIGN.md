# my-wine — 설계 문서 (Design Document)

> 내가 마신 와인을 사진으로 기록하고, 품종·지역별로 정리하며, 내가 낸 가격이
> 현지 시세 대비 적절했는지까지 판단해주는 **개인용 와인 기록 앱**.

이 문서는 구현에 들어가기 전, 합의된 설계 결정을 박제해 둔 기록이다.

---

## 1. 제품 개요

### 목적
- 와인 라벨을 사진으로 찍으면 품종·지역을 자동으로 정리해 기록한다.
- 마신 날짜, 함께 먹은 음식, 페어링 평가, 가격, 개인 평점을 함께 남긴다.
- 내가 낸 가격(소매/식당)이 **현지 시세 대비 적절한지** 판단해준다.

### 핵심 원칙
- **완전 무료 스택**: 유료 API에 핵심 기능을 의존하지 않는다. 외부 서비스가
  정책을 바꾸거나 막혀도 앱의 핵심 기능은 동작해야 한다.
- **개인 1계정 전용**: 다중 사용자 배포가 아닌, 본인 혼자 쓰는 개인 기록 앱.
  덕분에 하루 사용량이 적어(많아야 ~10병/일) 무료 한도 안에서 충분히 동작한다.
- **자동화 + 수동 보정**: 무료 OCR/매칭은 정확도가 100%가 아니므로, 자동 인식이
  실패하거나 틀리면 사용자가 직접 고칠 수 있는 폴백을 항상 둔다.

### 플랫폼
- **React Native** 모바일 앱 (iOS / Android)
- 데이터는 로컬 저장 중심 (개인 기록용)

---

## 2. 기술 스택 (완전 무료)

| 역할 | 선택 | 비용 | 비고 |
|---|---|---|---|
| 라벨 텍스트 추출(OCR) | Google **ML Kit Text Recognition** (on-device) | 무료, 키 불필요, 오프라인 | `react-native-mlkit-ocr` 등 |
| 품종/지역 기준 데이터 | **Wikidata SPARQL** (query.wikidata.org) | 무료, 키 불필요 | 품종·산지·AOC/AVA 계층 |
| 현지 시세(1순위) | **Wine-Searcher 무료 API** (Wine Check) | 무료 100콜/일 | 개인 사용 = 사실상 무료. 키 신청 필요 |
| 현지 시세(폴백) | **Vivino 비공식 스크래핑** | 무료 | 약관 그레이존, 소형 백엔드 필요, 깨질 수 있음 |
| 바코드 보조 매칭 | **Open Food Facts API** | 무료, 키 불필요 | 바코드 있는 와인 한정, 커버리지 들쭉날쭉 |

### 배제한 후보
- **wineapi.io / API4AI**: 일정 호출 이후 유료 → 완전 무료 원칙 위배.
- **Vivino 공식 API**: 존재하지 않음 (비공식 스크래핑만 가능).
- **클라우드 OCR(Google Vision / AWS Textract)**: 호출당 과금 → on-device ML Kit로 대체.

## 3. 데이터 모델

### Wine (와인 자체 — 라벨에서 식별되는 정보)
| 필드 | 설명 |
|---|---|
| `name` | 와인 이름 |
| `producer` | 생산자/와이너리 |
| `vintage` | 빈티지(연도) |
| `varieties` | **다중 품종 + 비율** 배열. 예: `[{grape: "Cabernet Sauvignon", percent: 60}, {grape: "Merlot", percent: 40}]` |
| `region` | **국가 › 산지 › 세부 AOC/AVA** 계층. 예: `{country: "France", region: "Bordeaux", subRegion: "Margaux"}` |
| `labelImage` | 라벨 사진 경로 |
| `referencePrice` | 조회된 현지 평균 소매가 (min/avg/max), 통화, 조회 출처, 조회 일시 |

### Tasting (마신 기록 — 한 번의 시음 이벤트)
| 필드 | 설명 |
|---|---|
| `wineRef` | Wine 참조 |
| `tastedAt` | 마신 날짜 (사진 EXIF 또는 수동 입력) |
| `purchaseType` | 구입처 유형: `restaurant` / `retail` |
| `pricePaid` | 내가 실제로 낸 가격 + 통화 |
| `foodPairing` | 함께 먹은 음식 (텍스트 또는 항목 리스트) |
| `pairingRating` | 페어링 평가: 좋음 / 보통 / 별로 |
| `tasteRating` | 맛 자체 평점 |
| `valueRating` | 가격 대비 가치 평점 (맛 평점과 **분리**) |
| `priceVerdict` | 적정가 자동 판정 결과(아래 §4): 저렴 / 적절 / 비쌈 |
| `notes` | 자유 메모 |
| `photos` | 추가 사진 |

> 한 와인을 여러 번 마실 수 있으므로 Wine : Tasting = 1 : N.

## 4. 적정가 판단 로직

조회된 **현지 평균 소매가**(`referencePrice.avg`)를 기준점으로 삼는다.

### 소매 구매(`retail`)
- 낸 가격을 현지 평균가와 직접 비교한다.
- 예: 현지 평균 4만원, 소매 구매 10만원 → 약 2.5배 → "비쌈".

### 식당(`restaurant`)
- 외식업계 표준 마크업은 **소매가의 2~3배**(등급별로 저가일수록 더 큼).
- 기준 범위 = `현지 평균가 × 2 ~ 현지 평균가 × 3`.
  - 범위 **이내** → "적절"
  - 범위 **초과** → "비쌈(바가지)"
  - 범위 **미만** → "저렴(득템)"
- 예: 현지 평균 4만원 → 식당 적정 범위 8~12만원. 20만원이면 "다소 비쌈".

### 시세 조회 폴백 순서
1. **Wine-Searcher 무료 API**로 조회.
2. 실패 시 **Vivino 비공식 스크래핑**.
3. 그래도 실패 시 **사용자가 현지가를 수동 입력** (마지막 안전망).
4. 끝내 시세를 못 구하면 `priceVerdict`는 "판단 불가"로 두고, 사용자 평점만 기록.

## 5. 인식 파이프라인 (사진 → 자동 정리)

```
사진 촬영
  → ML Kit OCR로 라벨 텍스트 추출 (생산자/빈티지/품종/지역명 등)
  → Wikidata 기준 사전과 퍼지 매칭하여 품종·국가·산지·AOC 자동 태깅
  → (바코드 보이면) Open Food Facts로 보조 매칭
  → Wine-Searcher / Vivino로 현지 시세 조회
  → 사용자 확인·보정 화면 (자동 인식 결과 수정 가능)
  → Tasting 입력(가격/음식/평점/날짜) → 저장
```

---

## 6. 알려진 트레이드오프

- **OCR 정확도**: 무료 on-device OCR은 화려한 라벨·필기체·곡면 병에서 인식률이
  떨어진다 → 수동 보정 UI가 필수이며, 이는 MVP에서 수용한다.
- **시세 커버리지**: 저가 마트 와인이나 비프랑스권 와인은 시세 매칭이 안 될 수
  있다 → 수동 입력 폴백으로 메운다.
- **Vivino 스크래핑**: 약관 그레이존이고 사이트 구조 변경에 취약 → 어디까지나
  폴백이며, 깨져도 핵심 기능은 무사하도록 인터페이스를 분리한다.
- **Wine-Searcher "trial"**: 영구 무료 보장 문서는 없으나 개인 사용량(하루 수 콜)
  으로는 잘릴 가능성이 낮다. 유료 전환이 필요해지면 시세 모듈만 교체할 수 있도록
  가격 조회 로직을 인터페이스 뒤에 둔다.

---

## 7. 향후 확장 여지 (현재 범위 밖)

- 유료 와인 DB(wineapi.io, Wine-Searcher 유료 티어)로의 무중단 업그레이드.
- 내 과거 기록 기반 자기 비교(같은 와인/품종+지역의 내 평균가 추이).
- 클라우드 동기화·다중 사용자(현재는 개인 1계정 전용).
